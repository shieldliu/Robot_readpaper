# Quick View

**Title**: BrowseComp-Plus: A More Fair and Transparent Evaluation Benchmark of Deep-Research Agent
**Authors**: Zijian Chen, Xueguang Ma, Shengyao Zhuang, Ping Nie, Kai Zou, Andrew Liu, Joshua Green, Kshama Patel, Ruoxi Meng, Mingyi Su, 等 (共20位作者)
**arXiv**: [2508.06600](https://arxiv.org/abs/2508.06600)
**Year**: 2025
**机构**: University of Waterloo, CSIRO, Carnegie Mellon University, University of Queensland

# Question

如何公平、透明、可复现地评估 Deep-Research Agent（深度研究智能体）的能力，同时能够独立分析检索器和LLM推理组件的贡献？

# Task

构建一个用于评估 Deep-Research Agent 的基准数据集，该数据集：
1. 提供固定的、人工验证的文档语料库
2. 支持检索器和LLM组件的独立评估
3. 消除对动态网络搜索API的依赖
4. 包含正样本文档和具有挑战性的负样本文档

# Challenge

1. **公平性问题**：现有评估（如BrowseComp）依赖黑盒网络搜索API，API结果动态变化且不透明，导致不同系统间无法公平比较，实验也难以复现
2. **透明度问题**：缺乏对文档语料库的控制，无法隔离检索器对最终性能的贡献
3. **可访问性问题**：商业网络搜索API成本高昂，且检索质量存在变异性
4. **语料库构建难度**：需要同时满足完整证据覆盖、检索难度适中、语料库规模实用三个相互制约的要求

# Insight

通过构建一个固定的、人工验证的文档语料库（包含支持文档和挖掘的困难负样本），可以将 Deep-Research 系统的端到端评估转化为可控的组件级分析，从而实现公平、透明、可复现的评估。

# Contribution

1. **BrowseComp-Plus 基准数据集**
   - **方法**: 从 BrowseComp 的 1,266 个问答对出发，使用 OpenAI o3 模型自动挖掘证据文档，再经过人工标注验证。最终得到 830 个查询、100,195 个文档的语料库。每个查询平均包含 6.1 个证据文档、76.28 个负样本、2.9 个 gold 文档
   - **技术优势**: 固定语料库使得不同系统可公平比较；人工验证确保文档质量；困难负样本提高检索难度

2. **两阶段证据文档收集流水线**
   - **方法**: 第一阶段用 o3 模型搜索支持答案的网页并提取线索-URL-证据三元组；第二阶段由人工标注员验证每个线索的文档支持，并标记包含最终答案的 gold 文档
   - **技术优势**: 结合自动化与人工验证，既高效又可靠；14名标注员共投入400+小时

![人工标注界面](figures/fig-006.png)
*图：人工标注界面，标注员验证每个线索的文档支持，并标记包含最终答案的 gold 文档*

3. **困难负样本挖掘方法**
   - **方法**: 用 GPT-4o 将复杂问题分解为平均7个子查询，每个子查询通过 Google Search API 获取最多100个搜索结果，抓取并处理这些页面作为困难负样本
   - **技术优势**: 保持语料库规模可控（~100K文档）的同时，确保检索任务足够困难

4. **系统性的检索器-智能体交互分析**
   - **方法**: 在受控条件下评估多种检索器（BM25、Qwen3-Embedding、ReasonIR）与多种LLM（GPT-5、o3、Opus 4、开源模型等）的组合
   - **技术优势**: 首次能够量化分析检索质量对 Deep-Research 系统的影响

# Experiments

## Core Contribution Impact (Ablation Studies)

### 检索器对端到端准确率的影响

![主要实验结果](figures/fig-000.png)
*图：不同LLM在BM25（红色方块）和Qwen3-Embed-8B（蓝色圆点）检索器下的准确率与搜索次数对比。更强的检索器（蓝色）在所有模型上都带来显著的准确率提升。*

| LLM | BM25 准确率 | Qwen3-Embed-8B 准确率 | 提升 |
|-----|------------|---------------------|------|
| GPT-5 | 55.9% | 70.12% | +14.2% |
| o3 | 49.28% | 63.49% | +14.2% |
| Sonnet 4 | 14.34% | 36.75% | +22.4% |
| Opus 4 | 15.54% | 36.14% | +20.6% |

**关键发现**：
- 更强的检索器不仅提高准确率，还**减少搜索调用次数**（GPT-5 从 23.23 次降到 21.74 次）
- 开源模型（Qwen3-32B、SearchR1-32B）在工具使用能力上显著落后闭源模型
- Oracle 检索（直接提供正确文档）下，gpt-4.1 达到 93.49% 准确率，证明检索是主要瓶颈

### 推理强度的影响（gpt-oss 系列）

| 模型 | 推理强度 | 准确率 | 搜索次数 |
|------|---------|--------|---------|
| oss-20B | low | 13.37% | 1.87 |
| oss-20B | high | 34.58% | 23.87 |
| oss-120B | low | 24.94% | 2.21 |
| oss-120B | high | 42.89% | 18.35 |

### 语料库规模的影响

将语料库从 100K 扩展到 ~10M 文档（添加 Fineweb-edu）：
- BM25 检索效果略有提升（IDF 估计更准确）
- 神经检索器效果下降（新文档未标注，被视为负样本）
- **结论**：100K 语料库足以支持有效的评估

## Limitation

![语料库统计](figures/fig-005.png)
*图：左侧为文档长度分布（中位数2336 tokens），右侧为覆盖答案所需的token数分布。512 token截断可覆盖86.5%的查询答案。*

1. **开源模型的工具使用能力不足**：Qwen3-32B 和 SearchR1-32B 平均搜索调用次数不足 2 次，远低于闭源模型的 20+ 次
2. **文档截断**：由于成本限制，只提供每个文档的前 512 tokens（86.5% 的查询仍能覆盖答案）
3. **Google Maps 类查询被排除**：42 个需要 Google Maps API 的查询被移除
4. **语料库为静态快照**：无法反映网页的动态更新
